今天跟大家分享一个最近遇到的，非常有意思的C语言“小坑”。短短几行代码，却藏着C语言底层运行机制的“终极秘密”！保证让你重新认识C语言！

### **1. 一段“简单”的C代码**

先给大家看一段“人畜无害”的C代码，你肯定觉得这代码简单到爆，一眼就能看穿：

```c
#include <stdio.h>

void foo(void)
{
	int a = 5;
	int b;
}

void bar(void)
{
	int x;
	printf("%d\n", x++);
}

int main(void)
{
	foo();
	bar();
	bar();
	
	return 0;
}
```

怎么样？是不是感觉像在看“Hello World” plus版？`foo()`函数里定义了两个int变量，`a`初始化为5，`b`没管它。`bar()`函数里定义了一个`x`，也没初始化，然后就直接`printf("%d\n",x++);`打印出来，还顺手`x++`自增了一下。`main()`函数里，先调一下`foo()`，再连续调两次`bar()`。问题来了：**这段代码运行后会输出什么？**

> 请各位大神预测一下，是“5和6”？还是“6和7”？或者干脆就是一堆“乱码”？又或者是，你觉得根本跑不起来直接报错？评论区留下你的答案，看看有多少小伙伴猜对了！

来让我们揭晓“正确答案”(注意，这里的“正确答案”要打引号，因为C语言的世界，很多时候并没有绝对的“正确”）。

### **2. “5和6”——看似合理的“期望答案”**

如果你跑了一下这段代码(还没跑过？赶紧跑一下试试！)，可能会惊喜地发现，它竟然输出了：

```bash
5
6
```

完美！跟我们“想当然”的结果一模一样！少年你还是太年轻了！C语言的“水”深着呢！这“5和6”的输出结果，其实只是特定条件下的“假象”而已！背后隐藏着一个叫做 **“未定义行为”(UndefinedBehavior)** 的“大坑”呢！

### **3. 栈——内存的“共享单车”**

想要搞清楚“未定义行为”是怎么回事，我们先要了解一下 **“栈”(Stack)**（*一种"后进先出"(LIFO)的内存结构。*）这个幕后功臣。你可以把“栈”想象成一块 **“内存共享单车停放点”**。每当一个函数被调用的时候，系统就会在栈上分配一块空间(叫做“栈帧”)，就像在停放点划出一块区域来停放自行车一样。这个栈帧用来存放函数的局部变量、参数、返回地址等等。我们的`foo()`和`bar()`函数里的`int a`,`int b`,`int x`这些局部变量，就是被安排在栈上的“共享单车”。函数返回时，栈帧被释放（实际上是栈指针移动），但物理内存中的数据并不会立即清除。

梳理一下这段代码的调用过程：

* **`foo()`函数的“骚操作”**：当`main()`函数调用`foo()`的时候，系统在栈上给`foo()`分配了一块栈帧。`foo()`函数里的`int a = 5;`就相当于在栈帧的某个位置放了一辆“共享单车”，并且给这辆车贴了个“5”的标签。`int b;`这行代码，只是在栈帧上又预留了一个位置，**并没有给它贴标签(没有初始化)**。当`foo()`函数执行完毕后，`foo()`的栈帧就被释放了，就像共享单车被骑走了一样。但是！但是！**重点来了！栈帧虽然被释放了，但是之前停放在那里的“共享单车”(内存数据)并不会立刻被清空！它可能还“残留”在那里！**

* **`bar()` 函数的“捡漏”行为**：接下来，当 `main()` 函数调用 `bar()` 的时候，系统又在栈上给 `bar()`分配了一块栈帧。`bar()` 函数里的 `int  x;`这行代码，也会在栈帧上预留一个位置，**同样没有初始化！** 但是！**神奇的事情发生了！`bar()`的栈帧，很可能正好 “重叠” 了之前 `foo()`用过的栈帧区域！** 也就是说，`bar()`新分配的 “停车位” ，很可能就是之前 `foo()`“停过车” 的地方！所以，`bar()`里的 `x`变量，就 “顺手牵羊”“捡漏”拿到了之前 `foo()` 留在栈上的 “残留数据”——也就是`a`的值 `5`！然后，`printf("%d\n", x++);`就打印出了`5`，再顺便把`x`的值自增变成了 `6`。

* **第二次`bar()`的“盲盒”**：第二次调用`bar()`的时候，栈的原理类似，但这次`x`“捡”到的值，就更像是在开“盲盒”了！可能“捡”到上次`bar()`留下的`x`值(自增后的`6`)，也可能“捡”到其他“垃圾数据”。总之，一切皆有可能！这就是 **“未初始化变量”的“威力”**！

### **4. 未定义行为——C语言的“薛定谔的猫”**

到这里，应该明白了吧？“5和6”的输出结果，其实是一种 **“偶然”** 现象，是程序在特定条件下，“碰巧”得到了这样的结果。**这种行为，在C语言中被定义为“未定义行为”(UndefinedBehavior,UB)**。你可以把“未定义行为”想象成C语言世界的“薛定谔的猫”，在程序运行之前，你永远不知道它会变成什么样！

* **“未定义行为”——C标准的“灰色地带”**：C语言标准只规定了“良好行为”(well-definedbehavior)，对于那些“不符合规范”的操作(比如，使用未初始化变量)，标准并没有强制规定编译器必须怎么做。这就像法律只规定了“好公民”应该怎么做，但对于“坏公民”会做出什么，法律并没有详细规定，而是留给了执法者一定的“自由裁量权”。C语言标准也是一样，它把“自由裁量权”交给了编译器实现者。

* **编译器的“任性”**：一旦你的代码触发了“未定义行为”，编译器就可以“为所欲为”了！它可以生成任何代码，程序可能会崩溃、跑飞、得到“意料之外”的结果，甚至看起来“一切正常”！**但你绝对不能依赖“未定义行为”来实现任何功能，因为它就像一颗“定时炸弹”，随时可能爆炸！**

### **5. 让“未定义行为”更加“疯狂”的操作**

如果你觉得“未定义行为”已经够可怕了，那还有更可怕的——**编译器优化！** 编译器优化就像“兴奋剂”，可以让程序跑得更快，但同时也可能让“未定义行为”更加“疯狂”！

* **优化——代码的“整容术”**：编译器为了提升程序性能，会施展各种“整容术”，对你的代码进行“魔改”。比如，发现`foo()`函数里的变量`a`和`b`没啥用，就直接把`foo()`函数整个“切掉”(优化掉)！再比如，为了提高速度，把栈上的局部变量“搬到”速度更快的**寄存器**里！这样一来，`foo()`可能根本不会在栈上留下任何“痕迹”，`bar()`自然也就“捡”不到`5`这个值了。

* **优化VS未定义行为：“混乱邪恶”组合**：一旦开启编译器优化，程序的行为就变得更加“不可预测”了。**你可能得到“乱码值”，也可能程序直接崩溃，甚至可能得到和未优化时一样的结果(但那也只是巧合！)**。总之，优化和未定义行为“狼狈为奸”，让C语言的“坑”变得更深更隐蔽！

### **6. 请关注编译器的 “唠叨”——警告信息！**

其实也别太悲观。现代编译器其实是很“善良”的，它会尽力帮助我们“避坑” 。**编译器会在编译过程中，对代码进行“体检” ，如果发现代码中可能存在“未定义行为”或其他潜在问题，就会发出“警告信息” (Warning)**，就像老母亲的“唠叨” 一样，提醒你要注意安全！

* **编译器警告——代码的 “体检报告”**：对于我们这段“糟糕” 的C代码，编译器通常会发出关于 **“未使用变量” (`-Wunused-variable`)** 和 **“未初始化变量” (`-Wuninitialized`)** 的警告。就像这样：

```shell
[build] [ 50%] Building C object CMakeFiles/MyExecutable.dir/main.c.o

[build] ~/c/main.c:5:9: warning: unused variable 'a' [-Wunused-variable]

[build] 5 | int a = 5;

[build] | ^

[build] ~/c/main.c:6:9: warning: unused variable 'b' [-Wunused-variable]

[build] 6 | int b;

[build] | ^

[build] ~/c/main.c:12:20: warning: variable 'x' is uninitialized when used here [-Wuninitialized]

[build] 12 | printf("%d\n", x++);

[build] | ^

[build] ~/c/main.c:11:10: note: initialize the variable 'x' to silence this warning

[build] 11 | int x;

[build] | ^

[build] | = 0

[build] 3 warnings generated.
```

* **重视警告，远离 “坑”**：**编译器警告，绝对不是 “耳旁风” ！它往往是代码中潜在bug的“预警信号”！** 我们应该像对待“圣旨”一样重视编译器警告，**养成“零警告” 编译的好习惯**，及时修复警告，把bug消灭在萌芽状态！这才是“优雅”的C语言编程姿势！
